## 1. Обрабатываемые данные и принцип минимизации (Data Minimization)

В основе архитектуры приложения лежит принцип минимизации собираемых данных (в соответствии с лучшими практиками GDPR и ФЗ-152). Система запрашивает и хранит только тот объем информации, который критически необходим для функционирования бизнес-логики.

**Данные, которые мы храним на сервере:**

- **Идентификационные:** Логин (уникальный псевдоним) и криптографический хеш пароля.
    
- **Финансовые:** Транзакции (сумма, тип, категория, дата, текстовый комментарий) и шаблоны регулярных платежей.
    
- **Юридические:** Флаг `consent_given` (булево значение), подтверждающий согласие на обработку данных.
    

**Данные, которые мы принципиально НЕ собираем:**

- Настоящие ФИО, номера телефонов и адреса электронной почты.
    
- Номера реальных банковских карт, CVC-коды или реквизиты счетов.
    
- _Примечание по UX/UI:_ Поле «Ваше имя», используемое для персонализации интерфейса фронтенда, сохраняется исключительно на стороне клиента (в браузерном `localStorage`) и никогда не передается на бэкенд. Это полностью исключает риск утечки персональных данных из базы данных.
    

## 2. Безопасность хранения данных (Data at Rest)

Инфраструктура хранения построена на базе реляционной СУБД **SQLite**, взаимодействие с которой осуществляется через ORM-систему **SQLAlchemy**.

**Механизмы защиты базы данных:**

- **Изоляция:** Файл базы данных (`finance.db`) находится в защищенном контуре Docker-контейнера. Доступ к нему извне невозможен, так как наружу «смотрит» только порт приложения FastAPI.
    
- **Защита паролей (алгоритм bcrypt):** Пароли пользователей никогда не хранятся в открытом виде. При регистрации пароль хешируется с использованием библиотеки `passlib` и криптографического алгоритма `bcrypt`.
    
    - _Обоснование выбора:_ `bcrypt` автоматически генерирует уникальную «соль» (salt) для каждого пароля, что делает невозможным взлом с помощью радужных таблиц (rainbow tables). Кроме того, алгоритм намеренно замедлен с точки зрения вычислений, что сводит на нет эффективность атак методом полного перебора (brute-force).
        

## 3. Безопасность сессий и передачи данных (Data in Transit)

Для аутентификации и авторизации пользователей реализован механизм **сессий без сохранения состояния** с использованием **JWT (JSON Web Tokens)**.

- **Жизненный цикл токена:** после успешной проверки логина и пароля сервер генерирует JWT-токен с криптографической подписью. Срок действия токена (Expiration Time) жестко ограничен **24 часами**.
    
- **Снижение рисков:** Ограниченный срок действия гарантирует, что даже в маловероятном случае компрометации токена злоумышленник потеряет доступ к системе максимум через сутки.
    
- **Хранение на клиенте:** Токен сохраняется во фронтенд-приложении (React) в `localStorage` и прикрепляется к заголовку `Authorization: Bearer` при каждом обращении к защищенным API-эндпоинтам (`operations`, `import`). Сервер проверяет подпись токена без необходимости дополнительных обращений к базе данных.
    

## 4. Векторы атак, оценка рисков и пути их устранения

При переходе от MVP-версии к полноценному коммерческому релизу (Production) необходимо учитывать следующие векторы атак:

### Риск 1: Перехват трафика (Man-in-the-Middle / MitM)

- **Угроза:** Поскольку в текущей среде разработки (dev-контейнер) данные передаются по протоколу HTTP, злоумышленник в той же публичной Wi-Fi сети теоретически может перехватить трафик и украсть JWT-токен или финансовую выписку.
    
- **Устранение (Production-ready):** При развертывании на реальном сервере весь трафик будет принудительно маршрутизироваться через протокол **HTTPS (TLS/SSL)**. Это обеспечит сквозное шифрование данных между браузером клиента и нашим сервером. Для реализации будут использоваться бесплатные сертификаты с автоматическим обновлением от _Let's Encrypt_ в связке с обратным прокси-сервером Nginx.
    

### Риск 2: Межсайтовый скриптинг (XSS)

- **Угроза:** Внедрение вредоносного JavaScript-кода на страницу пользователя, который может прочитать `localStorage` и украсть JWT-токен.
    
- **Устранение:** Фронтенд построен на React, который по умолчанию экранирует все вводимые данные (предотвращая внедрение). В качестве дальнейшего улучшения архитектуры безопасности планируется перенести хранение JWT-токена из `localStorage` в защищенные **httpOnly-куки**, которые физически недоступны для чтения через JavaScript.
    

### Риск 3: утечка данных через сторонние ИИ-сервисы

- **Угроза:** Передача конфиденциальной информации третьим лицам (провайдеру языковой модели).
    
- **Решение:** Модуль импорта ИИ обращается к отечественному API GigaChat. Во-первых, ключи аутентификации изолированы в файлах `.env` и не попадают в систему контроля версий (Git). Во-вторых, нейросети передается исключительно необработанный текст чека без привязки к личности (имя пользователя, логин или идентификатор в промпт не включаются). Таким образом, передаваемые данные полностью обезличены.
    

**Вывод:** Текущая архитектура приложения демонстрирует глубокое понимание принципов "Security by Design". Базовые уязвимости закрыты на уровне фреймворков (FastAPI, React), а работа с конфиденциальными данными выстроена с соблюдением требований к минимизации и криптографической защите.
